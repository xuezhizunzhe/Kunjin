# KunJin A 股情报与个人投资账本设计

日期：2026-07-11  
状态：已完成对话确认，等待书面规格 review

本设计扩展 `2026-07-11-kunjin-fund-research-assistant-design.md`。两份文档冲突时，以本设计对个人账本、基金资料补全、A 股情报和新闻研究的规定为准；既有的只读、安全、证据和非自动交易边界继续有效。

## 1. 背景与目标

KunJin 当前已经可以通过养基宝只读同步账户和持仓，保存正式净值，计算基础收益与集中度，并运行工作日定时同步。下一阶段要把它扩展为面向基金初学者的本地研究系统，覆盖 A 股和国内公募基金。

系统通过 Codex 接收自然语言问题。例如：

- “近两天有什么重要的金融市场新闻？”
- “未来 1 至 3 个月哪些基金方向值得继续研究？”
- “这只基金的经理、费用、季度持仓和同类表现怎么样？”
- “结合我的持仓，我应该继续观察、考虑补仓，还是优先控制风险？”

KunJin 必须先收集可核验数据，再做确定性计算和带证据的解释。它不能把近期上涨、媒体观点或单一评分包装成确定的投资建议，也不能操作支付宝、养基宝或执行交易。

## 2. 用户与默认研究框架

### 2.1 用户条件

- 用户处于基金学习初期，需要解释指标含义和决策风险。
- 投入资金未来 12 个月基本不会使用，并另有应急储备。
- 初期每月投入约 1,000 元，约为工资的十分之一。
- 随着知识和经验积累，月度预算可能增加，但系统不自动提高风险等级。
- 默认模式为谨慎的新手指导，不使用保证收益、抄底、必涨等措辞。

### 2.2 分层时间框架

每次市场或基金方向研究默认分成三个时间层：

1. **最近 48 小时**：新闻、公告、政策、突发事件、短期催化与即时风险。
2. **未来 1 至 3 个月**：价格趋势、市场宽度、资金变化、事件持续性、景气验证和拥挤度。
3. **未来 6 至 12 个月**：估值、盈利、产业结构、政策方向、基金质量和组合适配性，是主要投资判断层。

短期信息只能改变关注优先级，不能单独形成买卖结论。中期趋势用于验证市场是否持续认可某一逻辑。长期层负责判断该逻辑是否具有可持续基础。

## 3. 范围

### 3.1 本期范围

- A 股主要指数、行业、概念、ETF、个股基础行情与市场宽度。
- 国内公募基金的身份、份额类别、经理、费用、规模、基准、季度持仓、行业暴露、公告和正式净值。
- 基金同类分组、同类比较、持仓重合和候选基金筛选。
- 监管、交易所、基金公司、上市公司公告及可信财经媒体新闻。
- 养基宝当前持仓与支付宝截图交易证据形成的个人投资账本。
- 面向个人持仓的风险、重合、方向暴露和研究结论。
- 本地 SQLite 持久化、按来源降级、定时同步和问答时实时核验。

### 3.2 非目标

- 港股、美股、海外基金、期货、期权和加密资产研究。
- 自动登录或控制支付宝、手机养基宝 App、微信或浏览器。
- 自动购买、赎回、转换、调仓或定投。
- 从截图中读取密码、验证码、银行卡号等无关敏感信息。
- 以一个不透明综合分数代替分类研究。
- 预测短期涨跌或承诺收益。
- 第一阶段即迁移到 MySQL 或引入 Redis。

## 4. 总体架构

系统采用“本地定时情报库 + 提问时实时核验”的混合架构。

```text
用户问题 / 支付宝截图
          |
          v
Codex + kunjin-fund Skill
          |
          v
kunjin CLI 稳定 JSON 合约
          |
+---------+----------+----------+----------------+
| 个人投资账本       | 公募基金知识库          |
| 养基宝持仓         | 经理/费用/规模/基准     |
| 截图 OCR 与确认    | 持仓/行业/公告/同类     |
+--------------------+-------------------------+
| A 股市场与新闻情报 | 研究与对话决策层        |
| 行情/估值/资金/ETF | 三层时间框架/组合关联   |
| 公告/新闻/事件实体 | 正反证据/失效条件       |
+---------+----------+----------+----------------+
          |
          v
SQLite 规范化数据 + 脱敏原始快照 + 计算结果
```

Codex 负责识别问题、调用命令、必要时查询最新来源并向初学者解释。CLI 负责抓取、校验、存储和确定性计算。模型记忆不能用于填充经理、费用、持仓、交易或行情字段。

四个子系统必须保持独立接口：

- **个人投资账本**：回答“我持有什么、何时买入、证据有多可靠、实际成本是多少”。
- **公募基金知识库**：回答“这只基金是什么、由谁管理、收什么费用、持有什么、同类表现如何”。
- **A 股市场与新闻情报**：回答“市场发生了什么、哪些方向在变化、数据是否支持媒体叙事”。
- **研究与对话决策层**：把公共证据与个人组合关联，输出适合进一步研究的结论，不执行交易。

## 5. 数据来源与优先级

### 5.1 来源等级

1. **一级：正式披露与官方来源**
   - 中国证监会、上海证券交易所、深圳证券交易所、巨潮资讯。
   - 基金公司官网、基金合同、招募说明书、产品资料概要和定期报告。
   - 上市公司公告、指数公司正式编制方案。
2. **二级：结构化市场数据源**
   - 东方财富、天天基金等公开页面或接口。
   - 经审计后使用的 AkShare 数据源。
   - 每类数据设置主来源与备用来源，且保留实际来源。
3. **三级：可信财经媒体**
   - 新华社、证券时报、中国证券报、上海证券报等。
   - 报道、评论和机构观点必须保留作者或机构、发布时间和原始链接。

正式披露与平台数据冲突时，以正式披露作为研究主值，但必须保留并显示冲突。媒体观点不能覆盖事实字段。

### 5.2 按数据类型的来源策略

| 数据类型 | 主来源 | 备用来源 | 核验规则 |
| --- | --- | --- | --- |
| 基金身份、份额类别、状态 | 基金公司正式资料 | 结构化基金平台 | 六位代码、名称、A/C 类和生效日期一致 |
| 基金经理与任期 | 基金公告、定期报告 | 基金平台 | 保存精确起止日期，不把前任业绩归给现任 |
| 费用 | 产品资料概要、招募说明书 | 基金公司产品页 | 保存费率、适用条件、持有期限和文件日期 |
| 规模与份额 | 定期报告 | 基金平台 | 保存报告期和披露时间，不把旧规模称为当前值 |
| 业绩基准 | 基金合同、招募说明书 | 基金平台 | 保存基准文本、组成和有效区间 |
| 季度持仓 | 季报、半年报、年报 | 基金平台 | 保存报告期、披露日、完整性和占比口径 |
| 正式净值 | 基金公司或正式披露源 | 结构化基金平台 | 与盘中估值分表、分证据类型存储 |
| A 股行情与宽度 | 交易所或经审计结构化源 | 第二结构化源 | 交易日、证券代码、复权口径一致 |
| 估值、ETF 份额、资金 | 正式或经审计结构化源 | 第二结构化源 | 记录口径，禁止混合不可比序列 |
| 公告 | 交易所、巨潮、基金公司 | 无 | 以公告编号、主体和发布时间去重 |
| 新闻 | 官方媒体、可信财经媒体 | 其他具名来源 | 保留事实、引述和评论的区别 |

## 6. 个人投资账本

### 6.1 数据组合

- 养基宝自动同步当前账户、基金代码、基金名称、份额、市值、成本或收益等平台可观察字段。
- 支付宝截图提供申购、赎回、定投、分红等交易证据。
- 日常不要求重复提供养基宝截图；只有自动同步失败、代码名称冲突或用户需要补充平台未提供字段时才使用。
- 不尝试从手机 App 直接读取数据，也不控制支付宝或微信。

### 6.2 证据等级

- `transaction_confirmed`：支付宝交易详情或其他正式交易凭证可确认的记录。
- `user_confirmed`：OCR 结果经用户明确确认，或用户手工补录且确认准确。
- `position_inferred`：根据当前持仓、成本、份额等推导出的可能历史，不视为正式成交记录。

同一字段存在多个证据时，按 `transaction_confirmed`、`user_confirmed`、`position_inferred` 的顺序选择研究主值，同时保留其他记录和差异。

证据等级按字段判断，而不是给整张截图一次性定级。支付宝支付记录可以确认支付金额和支付时间，但如果页面没有基金代码、确认份额、确认净值或费用，就不能证明这些字段；它们仍保持缺失，或由其他证据单独确认。

### 6.3 截图导入流程

1. 用户在 Codex 对话中提供支付宝交易截图。
2. KunJin 通过独立 OCR adapter 在本机识别截图。macOS 第一实现使用 Apple Vision 框架及仓库内受测的本地 helper，不把图片上传给第三方 OCR 服务；Vision 不可用时退化为人工字段录入，不自动切换云 OCR。
3. 仅提取交易相关区域：基金名称或代码、交易类型、金额、份额、净值、费用、订单时间、确认时间、订单号后四位等。
4. 解析器按字段保存 OCR 文本、规范化值和置信度，不直接写入已确认交易表。
5. 对代码、交易方向、金额、时间或低置信度字段生成确认摘要。
6. 用户确认后写入 `transactions`；未确认内容保留在本地导入草稿中。
7. 系统将交易与养基宝持仓按基金代码、份额类别、时间和金额匹配，输出一致、可解释差异或待调查三种状态。
8. 导入时将截图复制到 `~/.local/share/kunjin/imports/` 并设置仅当前用户可读写。默认保留该私有副本，只有用户执行明确删除命令时才删除；用户原始图片永不由 KunJin 删除。数据库只保存受控副本路径、哈希和脱敏 OCR 结果。

第一阶段自动 OCR 模板只覆盖已经取得样例的支付宝申购或支付详情页。赎回、转换、现金分红和红利再投资先支持用户确认的结构化手工录入；取得脱敏样例并加入 fixture 后，再逐类启用自动 OCR，避免用一个宽松模板误识别所有页面。

### 6.4 对账规则

- 金额、份额、确认净值和费用采用十进制定点计算。
- 申购关系优先核验：`确认份额 * 确认净值 + 费用 ≈ 支付金额`。
- 当前持仓关系可核验：`当前份额 * 当前净值 ≈ 当前市值`。
- 平台平均成本只能作为观察值；没有逐笔记录时不能精确分解每批次盈亏。
- 赎回、转换、现金分红和红利再投资必须作为独立事件处理，不能用净持仓倒推出唯一历史。
- 金额差异容忍度由数据精度决定并在结果中显示，不能静默修正。

已确认样例必须可通过验收：基金 `519755`，支付宝支付金额 `20.00` 元、订单时间 `2026-07-04 23:11:51`；养基宝观察份额 `11.32`、平均成本 `1.7668`、当前金额 `19.77`、持有收益 `-0.23`。系统应显示 `11.32 * 1.7668 ≈ 20.00` 和 `11.32 * 1.7467 ≈ 19.77` 的一致性关系，并分别说明支付宝与养基宝支持了哪些字段，而不是把支付记录误称为基金确认单，或虚构缺失的确认净值和费用。

## 7. 公募基金知识库

### 7.1 主动补全规则

任何持仓分析、单基金研究或候选比较，只要出现六位基金代码，系统就应主动检查并按需同步：

- 基金代码、名称、类型、A/C 份额和存续状态。
- 当前及历史基金经理、精确任期和经理变更公告。
- 管理费、托管费、销售服务费、申购费、赎回费及适用条件。
- 基金规模、份额和变化。
- 业绩比较基准及有效区间。
- 最新季度持仓、报告期、披露日期和行业暴露。
- 同类分组、同类可比基金和比较结果。
- 与用户现有持仓的底层重合。
- 相关正式公告。

缺少实现不能被描述为“该信息不存在”。同步失败或正式来源未披露时，分别标记 `source_unavailable` 或 `not_disclosed`。

### 7.2 基金分类规则

- **指数基金**：指数编制方法、跟踪误差或差异、费率、规模、流动性和指数估值。
- **主动权益或混合基金**：经理任期内超额收益、风格稳定性、回撤、换手、集中度和规模变化。
- **行业或主题基金**：主题纯度、行业集中、周期位置、估值、拥挤和催化持续性。
- **债券基金**：久期、信用暴露、利率敏感度、回撤和流动性；数据不足时明确缺口。

### 7.3 同类比较

同类组不能只依赖平台标签。分组至少考虑基金类型、投资范围、基准、主动或被动、份额类别、成立时长和经理任期。A/C 类可共享底层组合，但费用和收益序列分别比较。

比较结果至少包含：

- 相同观察窗口内的收益、波动、最大回撤和恢复时间。
- 经理任期内表现，避免跨经理归因。
- 费用、规模、持仓集中度、换手和风格稳定性。
- 对当前组合的新增暴露与重合暴露。
- 数据覆盖率、同类组定义和幸存者偏差提示。

## 8. A 股市场、公告与新闻情报

### 8.1 市场实体

统一实体包括指数、申万或其他明确标准的行业、概念板块、A 股证券、ETF、公募基金、基金经理、基金公司、上市公司和政策机构。别名映射必须可追溯，不能仅凭名称相似自动合并。

### 8.2 板块六维研究

1. **趋势与宽度**：多周期收益、均线状态、上涨家数占比、创新高或新低分布。
2. **估值**：市盈率、市净率等适用指标及历史分位，注明亏损行业或周期行业的指标限制。
3. **盈利与基本面**：收入、利润、盈利预期或行业景气的可验证变化。
4. **成交、ETF 份额与资金**：成交量、换手、ETF 份额和可比口径资金序列。
5. **催化**：政策、产业、公司公告和宏观事件，并区分已发生事实与预期。
6. **拥挤与共识风险**：加速上涨、换手异常、估值扩张、主题集中和一致预期风险。

“近期强势”“基本面改善”“估值合理”“适合加入观察名单”是不同结论，不能互相替代。

### 8.3 新闻处理

- 抓取后按标题、正文指纹、公告编号和规范化链接去重。
- 保存发布时间和抓取时间，不用抓取时间代替发布时间。
- 从新闻中识别实体、事件类型、涉及时间和原文引述。
- 同一事件的多篇报道聚为事件簇，同时保留每篇来源。
- 官方公告属于事实证据；媒体转述需回链原始公告。
- 媒体或机构观点标记为 `market_opinion`。
- “新闻导致上涨”等因果关系默认标记 `reasoned_inference`，除非正式材料直接建立因果。
- 48 小时摘要按影响范围、证据等级、与当前持仓关联度和时效排序，不按情绪化标题排序。

## 9. 研究与结论合约

### 9.1 证据类型

- `verified_fact`：正式净值、公告、定期报告或同等来源直接支持。
- `deterministic_calculation`：可由已标识输入和参数重复计算。
- `reasoned_inference`：证据支持但不能唯一证明的推断。
- `market_opinion`：具名媒体、机构或人士的观点。
- `insufficient_data`：现有证据无法支持判断。

个人交易证据等级与研究证据类型是两个维度。例如，一笔支付宝申购既可是 `transaction_confirmed`，其金额字段又是 `verified_fact`。

### 9.2 允许的结论标签

- `worth_further_research`：值得进一步研究。
- `add_to_watchlist`：可加入观察名单，尚不构成交易建议。
- `insufficient_evidence`：证据不足。
- `recent_strength_high_chasing_risk`：近期强势但追高风险较高。
- `excessive_portfolio_overlap`：与当前持仓重合过高。
- `unfavorable_manager_or_fundamental_change`：经理或基本面出现不利变化。

每个重要结论必须同时包含：

- 支持证据与反对证据。
- 数据来源、原始发布时间或报告期、抓取时间。
- 适用的 48 小时、1 至 3 个月或 6 至 12 个月时间层。
- 与用户当前持仓、预算和风险暴露的关系。
- 结论失效条件。
- 数据缺口和下一步核验动作。

系统不输出一个通用、不透明的综合分。用户询问“继续持有、考虑补仓还是卖出”时，先输出事实、风险和条件式选择；除非证据充分且用户明确要求决策辅助，否则不把研究标签升级为直接操作指令。

## 10. 数据模型

在现有 `sync_runs`、`raw_snapshots`、`accounts`、`positions`、`funds`、`fund_nav`、`sector_snapshots` 和 `investment_theses` 基础上，新增下列逻辑实体。实际迁移按阶段拆分，避免一次引入全部表。

### 10.1 个人账本

- `imported_documents`：本地文件路径、哈希、文档类型、导入时间、保留策略。
- `ocr_fields`：字段名、原始文本、规范化值、置信度、区域和解析器版本。
- `transaction_drafts`：待确认的基金、方向、金额、份额、净值、费用与时间。
- `transactions`：确认后的交易事件、证据等级、来源文档和确认时间。
- `position_reconciliations`：交易账本与持仓观察的匹配状态、差异与计算版本。

### 10.2 基金知识

- `fund_share_classes`：基金代码、主基金关系、份额类别和有效区间。
- `fund_managers`：经理身份、任职起止、来源和披露时间。
- `fund_fees`：费用类型、费率或阶梯规则、适用条件、文件版本和有效区间。
- `fund_sizes`：报告期资产规模、基金份额和披露日期。
- `fund_benchmarks`：基准定义、组成和有效区间。
- `fund_holdings`：报告期、披露日期、证券代码、权重和披露完整性。
- `fund_industry_exposure`：报告期、行业标准、行业代码和权重。
- `fund_announcements`：公告编号、类型、发布时间、正式链接和关联实体。
- `peer_groups`、`peer_group_members`：分组规则版本和有效成员。

### 10.3 市场与新闻

- `market_entities`、`entity_aliases`：规范实体和来源别名。
- `market_series`：指数、板块、证券和 ETF 的带口径时间序列。
- `valuation_series`：估值指标、计算口径和历史分位所需序列。
- `flow_series`：成交、ETF 份额和资金流序列及口径。
- `announcements`：交易所、监管和上市公司公告。
- `news_items`：媒体、链接、发布时间、摘要和内容指纹。
- `news_events`、`news_event_links`：事件簇、实体、事件类型和文章关联。

### 10.4 研究结果

- `analysis_runs`：输入数据版本、参数、计算版本和状态。
- `analysis_evidence`：结论与正反证据的多对多关系。
- `research_conclusions`：时间层、标签、结论文本、失效条件和覆盖率。
- `watchlist_entries`：观察对象、加入理由、复查时间和状态。

原始数据、规范化事实、确定性计算和解释结论必须分层保存。金额、份额、净值、费率和权重使用十进制定点表示，时间保存带时区的 ISO 8601 值，交易日另存本地市场日期。

## 11. 同步、缓存与新鲜度

### 11.1 调度

- 现有工作日 18:30 的 `sync daily` 继续同步持仓、持有基金正式净值、市场收盘数据和应到期的基金元数据。
- 阶段 3 完成后，新闻和公告在每天 08:00、18:30 自动增量入库，覆盖周末政策与公告信息；提问时如果相关缓存超过 2 小时，再执行实时核验。
- 用户问题包含“今天、当前、最新、近两天”时，在回答前对相关数据类型执行实时核验。
- 季度持仓、经理、费用和基准按披露周期及缓存到期时间同步，不做无意义高频抓取。

### 11.2 新鲜度

每条观察至少保存：来源、来源发布时间或报告期、抓取时间、市场日期、数据类型和新鲜度状态。新鲜度阈值按数据类型定义，并使用 A 股交易日历，而不是简单工作日判断。

建议初始阈值：

- 行情与市场宽度：最近一个已收盘交易日。
- 新闻与公告：回答“近两天”时不超过 2 小时缓存，原始发布时间必须落在查询窗口内。
- 正式净值：最近一个应披露净值日。
- 经理和费率：最新有效正式文件，检测到新公告立即失效旧缓存。
- 季度持仓与规模：最新已披露报告期，始终显示报告期与披露日。

### 11.3 并发与存储演进

第一版继续使用 SQLite，并以 repository 和 storage interface 隔离持久化实现。当前是单用户、本机 CLI 和 launchd 串行任务，没有证据表明需要 MySQL 或 Redis。

只有出现可复现的并发写入、查询延迟或数据量瓶颈后才评估 MySQL。只有出现多个并发 worker 的速率限制、任务锁或共享短缓存需求后才评估 Redis。现有本地 Docker 容器不成为运行前置条件。

## 12. CLI 与 Codex 合约

保留现有命令，并逐阶段增加以下命令族：

```text
kunjin ledger import IMAGE...
kunjin ledger drafts
kunjin ledger confirm DRAFT_ID [--field NAME=VALUE...]
kunjin ledger add --type TYPE --fund-code CODE [transaction fields...]
kunjin ledger transactions [--fund-code CODE]
kunjin ledger reconcile [--fund-code CODE]
kunjin ledger document delete DOCUMENT_ID

kunjin sync fund-profile CODE...
kunjin sync fund-holdings CODE...
kunjin sync announcements [--entity ENTITY]
kunjin sync news [--since DURATION]
kunjin sync intelligence

kunjin fund profile CODE
kunjin fund fees CODE
kunjin fund holdings CODE [--period PERIOD]
kunjin fund peers CODE
kunjin fund compare CODE...

kunjin market overview
kunjin market sector NAME_OR_CODE
kunjin news recent [--hours 48]
kunjin research directions [--horizon short|medium|long]

```

所有供 Codex 使用的命令支持稳定 JSON envelope，至少包含：

```text
schema_version, command, as_of, data, sources,
freshness, evidence, warnings, conflicts, errors
```

错误采用稳定代码，不要求 Codex 解析自然语言错误。命令可以部分成功；单个数据源失败时，其他已成功部分仍返回。

## 13. 错误处理与降级

- 每个来源独立执行，东方财富某一板块接口失败不能阻断基金公告、净值或其他来源。
- 失败批次不得覆盖上次成功数据，旧数据保留并明确显示 `stale`、最后成功时间和失败原因。
- 主来源失败时才尝试已配置的备用来源；备用结果必须保留来源身份和口径。
- 解析失败保存脱敏诊断与解析器版本，不把半解析记录写入正式事实表。
- 身份、份额类别、日期或单位冲突时停止自动合并，记录 `data_conflict`。
- 正式披露未包含某字段时标记 `not_disclosed`；来源暂时不可用时标记 `source_unavailable`；历史不足时标记 `insufficient_history`。
- 新闻抓取失败不影响行情结论，但结论必须显示缺少事件证据。
- OCR 低置信度或对账不一致不创建确认交易，等待用户确认。
- 所有外部访问使用 HTTPS，禁止为了可用性降级到 HTTP。

## 14. 隐私与安全

- 养基宝令牌只保存在 macOS Keychain，不进入数据库、日志、快照、测试或 Git。
- 仅使用已审计的养基宝只读接口，不修改账户和持仓。
- 支付宝截图仅在本机读取；不使用第三方云 OCR。
- OCR 前后均避免记录完整订单号、身份证号、手机号、银行卡号或二维码内容；非研究字段应遮蔽或丢弃。
- 私有运行目录和截图目录限制为当前 macOS 用户访问。
- 日志、错误、原始快照和 JSON 输出统一经过敏感字段脱敏。
- 测试仅使用合成或明确脱敏的 fixture，不复制用户真实截图到 Git。
- 删除导入文档时只删除 KunJin 管理目录内的文件，不删除用户原始相册或聊天文件。
- 系统不保存 Codex 对话全文，只保存用户明确确认的交易、论点或观察名单信息。

## 15. 测试策略

### 15.1 单元测试

- OCR 字段规范化、置信度阈值和交易方向识别。
- 金额、份额、净值、费用、赎回和分红对账。
- 经理任期切分、费率阶梯、A/C 类关系和报告期处理。
- 同类分组、持仓重合、行业暴露、收益、回撤和恢复计算。
- 板块六维指标、估值分位、市场宽度和三层时间窗口。
- 新闻去重、事件聚类、实体映射和证据标签。

### 15.2 合约与集成测试

- 每个 adapter 使用脱敏 fixture 的解析合约测试。
- 主来源失败、备用来源成功、全部失败和过期缓存场景。
- SQLite 迁移、事务回滚、重复同步幂等和并发任务锁。
- CLI JSON schema、部分成功、冲突和错误代码 golden tests。
- launchd 首次同步、重试和状态可见性。
- 凭证、订单信息和截图路径不泄漏测试。

### 15.3 真实冒烟测试

- 使用现有养基宝授权同步真实持仓，但输出和日志不得暴露令牌。
- 用用户明确批准的支付宝截图完成一次 OCR、确认和对账流程。
- 对至少一只真实持有基金核对基金公司正式资料、经理、费用和最新季度持仓。
- 从主、备用来源各验证一个 A 股市场数据类型。
- 拉取最近 48 小时的真实公告或新闻并回链正式来源。
- 运行一次 `sync daily` 和一次完整研究问答，验证部分来源失败时仍可解释降级结果。

## 16. 分阶段交付与验收

### 阶段 1：支付宝截图与个人成本账本

- 使用本地 Apple Vision 导入支付宝申购或支付详情截图，并生成带置信度的交易草稿。
- 用户确认后形成不可静默修改的交易记录。
- 自动 OCR 支持申购或支付详情；赎回、转换、定投、现金分红和红利再投资先支持结构化手工确认录入。
- 与养基宝持仓对账并显示证据等级、差异和缺失字段。
- `519755` 的 20 元样例可以被正确解释且不虚构费用或确认净值。

### 阶段 2：基金正式资料与同类比较

- 按基金代码主动获取经理、费用、规模、基准、季度持仓和公告。
- 正确处理 A/C 类、经理任期和披露滞后。
- 对持有基金生成同类比较和底层持仓重合分析。
- 任一字段都可以追溯到正式来源或明确的备用来源。

### 阶段 3：A 股与新闻情报

- 建立指数、板块、ETF、证券、公告和新闻实体数据。
- 支持估值、宽度、成交、ETF 份额、可比资金序列和事件聚合。
- 东方财富单接口失败时可使用缓存或备用来源，不影响其他同步。
- 最近 48 小时摘要区分事实、观点和推断。

### 阶段 4：分层研究与组合关联

- 对方向研究分别输出 48 小时、1 至 3 个月和 6 至 12 个月判断。
- 使用板块六维框架，并按基金类型选择不同分析规则。
- 输出正反证据、来源时间、个人组合重合和失效条件。
- 只使用允许的结论标签，不生成通用不透明评分或自动交易指令。

### 阶段 5：调度、降级与端到端验证

- 扩展 `sync daily` 与 launchd，且任务状态可通过 `kunjin status` 检查。
- 使用 A 股交易日历和按类型新鲜度阈值。
- 完成真实持仓、正式基金资料、公开市场、公告新闻和截图账本冒烟测试。
- 更新 README、Skill 命令和限制说明。
- 全部测试通过，敏感信息扫描无结果，失败来源有明确降级输出。

## 17. 实施顺序与边界

实现严格按五个阶段推进，每阶段先写迁移与接口测试，再实现 adapter、service、CLI 和 Skill 集成。阶段验收通过后再进入下一阶段，避免同时扩张个人账本、公共数据和研究逻辑造成无法定位的数据错误。

第一阶段不需要 Docker、MySQL、Redis、Web UI 或 MCP server。SQLite repository 接口、稳定 JSON 合约和可替换 adapter 是后续扩展点。只有真实瓶颈或新的交互需求成立时，才单独设计基础设施或界面演进。

本设计不修改现有只读边界：KunJin 可以帮助用户理解“继续观察、考虑增加投入或优先降低风险”的条件，但最终交易由用户在自己的平台中手工完成。
